# WaniKani iOS Agent Guide

Use this file as the working ruleset for contributors and AI agents.

## 1) Project

- Native SwiftUI app using WaniKani API v2.
- Priorities: performance, reliability, modularity, modern Swift concurrency.

## 2) Tooling

- Xcode project is generated by `xcodegen`.
- Do not edit `.xcodeproj` directly.

| Command | Action |
|---|---|
| `make generate` | Regenerate Xcode project after file adds/removals. |
| `make build` | Build `WaniKani` scheme for iPhone 16 simulator. |
| `make test` | Run all unit and UI tests. |
| `make open` | Open generated project in Xcode. |
| `make clean` | Remove build artifacts and generated `.xcodeproj`. |

## 3) Structure

- `WaniKani/` (app layer)
- `WaniKani/App/`: app entry + root navigation.
- `WaniKani/Features/`: feature modules (`Views`, `ViewModels`).
- `WaniKani/Shared/`: shared UI, extensions, theme.
- `WaniKaniCore/` (business logic)
- `WaniKaniCore/Networking/`: API clients/endpoints/errors.
- `WaniKaniCore/Models/`: Codable API models.
- `WaniKaniCore/Persistence/`: storage integration.
- `WaniKaniCore/Security/`: keychain/session/auth.
- `WaniKaniCore/Managers/`: domain managers (sync, notifications, etc.).

## 4) Architecture

- Strict MVVM.
- Views stay passive and observe ViewModels.
- All ViewModels must be `@MainActor`.
- Use `@StateObject` for ownership, `@ObservedObject` for injection.
- Use repository protocols (`SummaryRepositoryProtocol` style).
- ViewModels depend on protocols, not concrete repository implementations.

## 5) Swift Concurrency

- Strict concurrency checking is expected.
- UI-facing types: `@MainActor`.
- Core shared mutable state: prefer `actor`.
- Prefer `async/await` and structured concurrency over completion handlers.
- Cross-actor transferred models should conform to `Sendable`.

## 6) Coding Standards

- Naming: `PascalCase` for types, `camelCase` for variables/functions.
- Prefer descriptive names over short/ambiguous names.
- Use standard Swift formatting (4-space indentation).
- Use `WaniKaniError` for domain errors.
- Avoid force unwraps (`!`) outside justified cases (tests/constants).
- Surface user-facing failures gracefully.
- Public `WaniKaniCore` APIs should use DocC comments (`///`).
- Explain the reason behind complex logic.

### Logging

- Use `SmartLogger.shared`.
- Keep logs inside `#if DEBUG`.
- Never log sensitive data (tokens, emails) in plaintext.
- Use correct log levels: `debug`, `info`, `error`, `fault`.

## 7) Testing

- TDD is encouraged.
- Add unit tests for every ViewModel and repository.
- Use protocol-based mocks.
- Test naming format: `test_[functionName]_[scenario]_[expectedOutcome]`.
- Cover core logic and edge cases.

## 8) Git Workflow

- `main`: stable, PR merges only.
- `feature/short-description`
- `fix/short-description`
- All code changes must go through a PR.
- Keep commits atomic (one logical change per commit).
- Match existing repo commit style (semantic/plain).
- Include author attribution for automated commits.
- PRs must pass CI and include clear summaries (plus UI screenshots when relevant).

## 9) App Identity

- App identity/build settings live in `project.yml`.
- To rename app/bundle ID: edit `PRODUCT_BUNDLE_IDENTIFIER` in `project.yml`, then run `make generate`.

## 10) WaniKani API v2

- Base URL: `https://api.wanikani.com/v2`
- Header: `Wanikani-Revision: 20170710`
- Auth: `Authorization: Bearer <token>`
- Rate limit: 60 req/min; handle `429 Too Many Requests`.
- Docs: <https://www.wanikani.com/api/v2>

## 11) Agent Execution Rules

- Plan first for non-trivial tasks.
- Keep files in the correct feature/core module.
- Run `make build` and `make test` after structural or logic changes.
- Run `make generate` after adding/removing files.
- Do not reintroduce deprecated WebView/Hybrid prototypes.

## 12) Completion Reporting

Every completion report must include this section:

**Technical Analysis:**
- **Problem**: [technical root cause]
- **Solution**: [specific code change]

Example:

> **Problem**: Launch crash caused by missing `WaniKani/Shared` in `project.yml` sources.
> **Solution**: Added `WaniKani/Shared` to `sources` and regenerated the project.

## 13) Pencil Rules

- Design file: `apps/ios/designs/wanikani-master.pen`
- For `.pen` files, use only `mcp__pencil__*` tools (never Read/Grep/Write).
- Run Pencil MCP operations in Task-spawned agents only (`subagent_type: general-purpose`), not the orchestrating session.
- After every `batch_design` or `set_variables` file change, save with `Cmd+S` via AppleScript.

Example:

```bash
osascript -e '
tell application "Pencil"
    activate
end tell
delay 0.5
tell application "System Events"
    tell process "Pencil"
        keystroke "s" using command down
    end tell
end tell
'
```

## 14) Git Push Policy

- Never `git push` unless the user explicitly asks. Commit only.
